# 第 14 章 生产工程化

你的 Agent 在 Jupyter Notebook 里跑得很顺畅——响应快、效果好、Demo 惊艳。然后老板说"下周上线"，你才发现从原型到生产之间隔着一道鸿沟。

生产环境里，单用户变成了数千并发用户，偶尔崩溃变成了 99.9% 的 SLA 承诺，"无所谓"的 API 费用变成了每月数万元的成本压力。本章就是帮你跨过这道鸿沟的桥梁。

我们会从最基础的项目结构和环境变量管理开始，教你把一个 Python 脚本变成一个可部署的服务；然后进入架构设计和容错体系，让你的服务在各种异常情况下都能优雅应对；最后深入成本控制、高可用架构和自动化发布流程，打造真正的生产级系统。

## 本章结构

| 层级 | 内容 | 适合读者 |
|------|------|---------|
| [入门篇](./beginner.md) | 项目结构、环境变量管理、基本错误处理、日志规范 | 第一次部署 Agent 服务的开发者 |
| [进阶篇](./intermediate.md) | 架构设计、完善的容错体系（重试/降级/熔断）、容器化部署 | 需要保证服务稳定性的工程师 |
| [高级篇](./advanced.md) | 成本控制、高可用架构、灰度发布、CI/CD、自动扩缩容 | 负责大规模 Agent 服务的架构师 |

## 从原型到生产的关键差异

| 维度 | 原型阶段 | 生产阶段 |
|------|---------|---------|
| 并发 | 单用户交互 | 数千并发用户 |
| 可靠性 | 偶尔崩溃可接受 | 99.9% SLA 承诺 |
| 成本 | 无预算限制 | 每请求成本敏感 |
| 安全 | 信任所有输入 | 防注入、防滥用 |
| 可观测 | print 调试 | 结构化日志 + 分布式追踪 |
| 部署 | 本地运行 | Docker + CI/CD 自动化 |
| 配置 | 硬编码 | 环境变量 + 配置中心 |

## 生产化路线图

一个 Agent 从 Demo 到生产，通常需要经历这些阶段：

```
阶段1 - 脚本可用（入门篇）
├── 标准项目结构
├── 配置管理（.env + Pydantic Settings）
├── 基础错误处理
├── 结构化日志
└── FastAPI HTTP 接口

阶段2 - 服务可靠（进阶篇）
├── 分层架构设计
├── 完善容错（重试 / 超时 / 熔断 / 降级）
├── 数据库集成（SQLAlchemy + Redis）
├── 异步任务队列
├── Docker 容器化
└── CI/CD 管道

阶段3 - 规模可扩（高级篇）
├── 高可用架构
├── 灰度发布策略
├── 成本优化体系
├── 自动扩缩容
├── 多租户隔离
└── Kubernetes 编排
```

## 你将学到的核心技能

- **入门篇**：把脚本变成可部署的 HTTP 服务，有配置管理、错误处理和日志
- **进阶篇**：构建分层架构，实现容错体系和容器化部署，让服务稳定可靠
- **高级篇**：掌握高可用、灰度发布和成本优化，支撑大规模 Agent 平台

## 常见的生产事故（以及如何避免）

| 事故类型 | 原因 | 预防措施 |
|---------|------|---------|
| API Key 泄露 | 硬编码在代码中 | 环境变量 + Secret Manager |
| 雪崩式故障 | 一个服务挂了拖垮整个系统 | 熔断器 + 降级策略 |
| 账单爆炸 | 死循环调用 LLM | Token 上限 + 费用告警 |
| 数据丢失 | 没有持久化 | 数据库 + 定期备份 |
| 服务不可用 | 单点故障 | 多实例 + 负载均衡 |

::: tip 工程化不是一蹴而就的
不需要一上来就搭建完美的架构。从入门篇开始，先让服务能跑起来，再逐步加固。过度工程化和工程化不足一样有害——前者浪费时间，后者出了事没法修。
:::
