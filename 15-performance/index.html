第 15 章 性能优化 ​

用户给 Agent 发了一条消息，等了 8 秒才看到回复——这种体验是不可接受的。性能优化是 Agent 从"能用"到"好用"的关键一步。

Agent 系统的性能挑战和传统 Web 服务不太一样。传统服务的瓶颈通常在数据库查询和计算，Agent 的瓶颈主要在三个地方：LLM 推理延迟（500ms-5s 每轮）、工具执行耗时（网络请求、数据库查询）、以及多轮迭代的累积效应。一次用户请求可能触发 3-5 轮 LLM 调用，每轮都带着越来越长的上下文。

本章从延迟分析入手，先搞清楚时间花在了哪里；然后介绍缓存策略和并发控制等实战技术；最后深入语义缓存、智能模型路由和降级链等高级优化，让你的 Agent 既快又省。

本章结构 ​

层级内容适合读者
入门篇延迟来源分析、Streaming 降低感知延迟、基本异步调用想让 Agent 响应更快的开发者
进阶篇并发控制、缓存策略（Prompt Caching + 语义缓存）、模型路由优化多用户并发场景的工程师
高级篇预测性执行、Fallback 降级链、连接池优化、端到端延迟优化追求极致性价比的架构师

Agent 延迟的典型构成 ​

用户请求 → 上下文准备(~200ms) → LLM 首 Token(~500-2000ms)
→ Token 生成(~1-5s) → 工具执行(~100ms-5s)
→ LLM 第二轮(~500-2000ms) → 响应传输(~5ms)
总延迟: 2-15 秒

优化策略全景 ​

不同层级的优化手段解决不同的问题：

优化层级手段效果实施成本
感知优化Streaming 输出感知延迟 -80%低
并发优化异步调用、工具并行吞吐量 2-5x低
缓存优化Prompt Caching、语义缓存延迟 -50%、成本 -60%中
路由优化简单问题用小模型成本 -70%、延迟 -60%中
架构优化预测执行、降级链P99 延迟 -50%高

性能优化的基本原则 ​

1. 先度量，再优化

不要凭感觉猜瓶颈在哪。给每个阶段加上耗时追踪，用数据说话。很多时候你以为的瓶颈不是真正的瓶颈。

2. 优化收益最大的环节

LLM 推理占了 60-70% 的延迟，优化它的 ROI 远高于优化网络传输。把有限的精力花在刀刃上。

3. 用户感知比实际延迟更重要

Streaming 不会减少总耗时，但用户从"等 5 秒看到结果"变成"0.5 秒就开始看到文字流出"，体验提升巨大。

4. 权衡成本和速度

更大的模型更准但更慢，缓存省钱但可能返回过时结果。找到你的业务能接受的平衡点。

你将学到的核心技能 ​

  • 入门篇：用延迟追踪器定位瓶颈，实现 Streaming 输出，掌握异步并行调用
  • 进阶篇：实现语义缓存和 Prompt Caching，构建并发控制体系，实现模型路由
  • 高级篇：设计预测性执行策略，构建 Fallback 降级链，进行端到端延迟优化

过早优化是万恶之源

Donald Knuth 说的。先让你的 Agent 正确工作，再考虑性能。很多性能问题可以通过简单的 Streaming + 异步调用解决 80%，不需要上来就搞复杂的缓存和路由。
